Overview

So one day, I wanted to improve my programming and math skills. I searched for problems to solve using a programming language and I stumbled upon ProjectEuler.net and I though it was the perfect problem source.

If you check the code, you will notice that it is divided in two "areas", the first is the Project Euler problems "solver", the second is the infrastructure that allows you to call the problems and display the result (aka UI).

At a previous iteration, the program used QT for its UI, but then, at the first portuguese Comicon I was testing the game of a friend and gave him a suggestion for a new feature, and he said that it would take to long to implement and I thought "too long, it takes like 10 minutes on Unity", then he told me that he was making the engine and that the feature would take too long to implement.

So I started to think about the problems of implementing an engine(at least a small/simple one) and then decided to try my luck. I dropped all QT code and started working on a new UI using SDL2 with the final goal of creating a small puzzle game based on games like Electric Box (which will, also, allow me to study more level design).


How to compile the program

The development of the program is being done on a Linux system (fully updated Archlinux) using g++ as the compiler and GDB as the debugger. Other than the standard c++ libs, it used SDL2, SDL2_ttf, SDL"_image, boost::multiprecision lib and GMP(linux only).

The boost::multiprecision lib is used because it let us change its backend lib. On linux it used GMP and on windows it used the "native" boost implementation (cpp_int) and the selection is made with preprocessor directives in the header file "defines.hpp". This approach was used as I was unable to correctly compile GMP and MPIR on windows.

Linux

On linux you can use the provided Makefile to compile the program and you need to have installed gcc, make, SDL2, SDL2_image, SDL2_ttf, boost and GMP. For easier instalation use your distro package manager to download and install the libs. The compiled program will be inside the "bin" folder.

Windows

The provided windows build was compiled using VS Express 2015. For you to compile it yourself, first you need to download the needed libs:

  • SDL2 (dev libs)
  • SDL2_image (dev libs)
  • SDL2_ttf (dev libs)
  • boost

After you download the libs create a new vs c++ project (I created an empty one and used "mainCRTStartup" as the linker entry point) and add the program source files (you need to add the lib/jsoncpp folder to your include search path). Add the include dirs for all the downloaded libs to your project include search path as well as add the 3 SDL libs "lib" subfolders (I used the x64 for all 3) to your lib search path. Finally your need to add SDL2.lib, SDL2main.lib, SDL2_ttf.lib and SDL2_image.lib to your linker input. The boost::multiprecision lib does not need any config from the linker as it is a header only lib.

Now, you should be able to compile the program on windows (if you did not add the .dlls to your system or their path to environment path var, you need to put the .dlls next to the exe, you can find the .dlls in the lib folders for all SDL libs). In case you are unable to correctly configure the SDL libs check this tutorial for them.

Folder and Files (Server)

  • 'src/' - contains the entire source code
  • 'src/lib/' - contains external downloaded libs which source code will be compiled into the program (jsoncpp in this case)
  • 'src/include/' - all the header files
  • 'src/src/' - all the .cpp files. main.cpp is in 'src' and not this subfolder
  • 'docs/' - documentation generated by doxygen
  • 'bin/' - destination (with the correct data folder structure) for the build compiled by using the Makefile
  • 'objects/' - folder that store the object files generated during compilation
  • 'euler_short' - small bash script to run the program after compilation
  • 'Makefile' - default file used by make to compile the program (in the future I need to improve/simplify this file)
  • 'readme_boost.html' - this file

The application

For this application I chose to use C++ (c++11/14) and use the maximum of its native and stl(like threads, clock and unique_ptr) functionalities before relying on external libs. For that reason I am only using SDL2, SDL2_image, SDL2_ttf, GMP(not on windows), jsoncpp and boost::multiprecision as external libs (all cross-platform, except for the problems with GMP). The 'src' folder contains the source code, 'docs' folder contains the output of doxygen for this problem and the 'bin' folder has a linux build in there. The euler_short script is just a shortcut to run the program if you are using a terminal to build it (on linux).

The comments used to generate the doxygen files, are present in the header files only, cpp files comments are only notes regarding the implementation.

Now I'll give a brief description about some of the files that contain the UI code.
poker_hand.cpp/.h, utils.cpp/.h, graphs.h, eulerproblems.h and every file inside the problems folder contain the implementation of the problems and their support functions. Keep in mind that things are being implemented as I need them, so some classes may look very incomplete.

main.cpp

Where it all starts, the program entry point. Owns the main state machine for the problems and the main loop that reads inputs, calls logic state function and renders the scene, also controls SDL init and termination

button.cpp/.h

Class to display a generic button

clock.cpp/.h

Basic SINGLETON class to manage a small clock to be mainly used by Euler Problems functions to time their execution

configparser.cpp/.h

Tries to load(from file) application configuration into config_json var

message_writer.cpp/.h

Singleton class controlling a basic threaded output writer to the console and/or output box. Currently the output to the UI is disabled due to some problems with texture corruption

object.cpp/.h

Base class for any element, to be like GameObject in Unity3d (although its far from it, for now)

problems_menu.cpp/.h

One of the states used by the main.cpp state machine. It lets the user call for problems and displays their result.

splash_screen.cpp/.h

One of the states used by the main.cpp state machine. For now its just a simple splash screen before the problems state.

state_machine.cpp/.h

Simple state machine that works with stateinterface objects

text.cpp/.h

Class to render text on screen. Currently not suited for displaying dynamic text because for each change causes a new texture to be created and the previous to be destroyed. I am, already, working on a solution based on spritesheet font.

texture.cpp/.h

Class that takes care of loading and storing a texture to be rendered

structs.h

Defines multiple structs that are used by

stateinterface.h

Basic Interface to be used by the main state machine

input_interface.h

Basic Interface to add behaviours to elements

renderinterface.h

Basic Interface to add render behaviours to elements

The application

Currently the following problems are being worked on 26,47,61,72,82 and 94

I will only speak about 72 and 94 as they are in a "special" stage, the others are either incomplete or have some error that I have yet to find.

Both 72 and 94 are solved, at least mathematically, and both suffer from the same problem floating point precision.
For 72 the solution passed by a sum of phi (Euler's totient function) for 1 < n < 1000000001 and using wolframalpha.com. When the upper limit ≤ 100000000 (100 millions) equals the one from wolphram, when the limit is the on of the problem (1 thousand of millions) the value misses My analysis takes me to believe that when the prime number used by 'phi' is to big it causes precision error.
For 94 the problem found is that some values are losing the decimal part, that is, some result, if calculated by hand give a result with decimal, but the algorithm cuts it and return only the integral part, breaking the algorithm validation code.
